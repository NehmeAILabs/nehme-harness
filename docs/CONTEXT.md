# Context Mode

Context mode is a SQLite-backed externalization and retrieval layer that keeps
large tool outputs out of the live context window while making them searchable
on demand. It is enabled by default (the `context` Cargo feature).

## Overview

When the agent runs a tool (bash, read, web_fetch, subagent) that produces a
large output, context mode:

1. **Stores** the full output in a SQLite database (`context.db`)
2. **Indexes** it with dual FTS5 tokenizers for full-text search
3. **Returns** a compact pointer (`id:N`) to the agent instead of the full output
4. Makes the full output **retrievable** on demand via `ctx_retrieve`

This reduces context window consumption significantly. On benchmark tasks,
context mode reduces context window usage by ~28% compared to baseline.

## How It Works

### Auto-externalization

Tool outputs are automatically externalized when they exceed thresholds:

| Source | Threshold | Behavior |
|--------|-----------|----------|
| bash output | >100KB | Full output indexed, returns first 2KB + pointer `[Output externalized: id:N]` |
| bash output | >2KB saved by compression | Silent indexing (no pointer shown) |
| read with intent | >5KB | Intent-filtered, full output indexed |
| web_fetch | large pages | Full content indexed |
| subagent response | >100KB | Returns 8KB preview + `[Subagent output externalized]` pointer |
| ctx_execute | always | Output compressed and indexed automatically |

The pointer format is:
```
[Output externalized: id:42 150000 bytes from `cargo test`. Use ctx_retrieve("42") to get full content. Dropped: 150 lines: 3 error lines, 12 warnings, 200 test results (12 failures)]
```

The `Dropped:` summary is generated by `summarize_dropped_content()` which
categorizes the dropped content (error lines, warnings, test results, functions,
imports) so the agent knows what it's missing without reading it.

### Silent indexing

Even when output is small enough to return in full (>2KB saved by compression),
the full output is silently indexed. This means `ctx_search` can find content
from outputs that the agent already saw but may have forgotten after compaction.

### Compact snapshots

On `/compact` (or `/compress`), a priority-bucketed snapshot of the session is
saved to SQLite:

| Priority | Content | Examples |
|----------|---------|----------|
| **P1** (must keep) | System messages, errors, decisions | `error`, `fail`, `blocked`, `todo`, `fixme`, `task:`, `decided`, `decision:` |
| **P2** (should keep) | File changes, test results | `wrote`, `created`, `modified`, `edited`, `deleted`, `test`, `build`, `passed` |
| **P3** (nice to keep) | Short messages | <500 chars, excluding greetings |

The snapshot is capped at 8KB. On session resume, the most recent snapshot is
loaded and injected into the system prompt under `## Previous Session Context`.

This ensures critical state (errors, decisions, file changes) survives
compaction deterministically, rather than depending on the model's summary.

## Tools

### `ctx_search`

Full-text search of all indexed tool outputs and events from the current session.

```json
{
  "query": "test failure auth module",
  "category": "tool_result",
  "limit": 5
}
```

- **query**: search terms (supports stemmed word search + substring matching)
- **category** (optional): filter by event category (`tool_call`, `tool_result`, `command_fail`, `file_write`, `file_edit`, `file_read`, `compaction`, `permission_deny`)
- **limit** (optional, default 5): max results

**Throttling**: After 3 calls, result previews are reduced to 200 chars. After 8 calls, the tool returns a "throttled" message.

**Permission**: read-only, always allowed.

### `ctx_execute`

Execute code in a subprocess with automatic output compression and indexing.

```json
{
  "language": "python3",
  "code": "import os; print(os.listdir('.'))",
  "intent": "list files"
}
```

- **language**: `python3`, `node`, `bash`, `ruby`, or `go`
- **code**: the code to execute
- **intent** (optional): if provided and output >5KB, only lines matching intent keywords are returned

The full output is always indexed in the context store. The returned output is
either intent-filtered, compressed, or raw (depending on size).

**Permission**: treated like bash — requires permission in guarded/standard modes, denied in readonly mode, allowed in yolo mode.

### `ctx_retrieve`

Retrieve the full raw output of a previously externalized tool result.

```json
{
  "query": "42"
}
```

- **query**: an ID number (e.g. `"42"`) or a keyword. If a number, fetches by rowid. If a keyword, fetches the most recent tool_output whose path_or_cmd or summary matches.

Returns up to 8KB of the raw output with a `[Retrieved: {tool} from {source}]` header.

**Permission**: read-only, always allowed.

### `ctx_stats`

Show context savings statistics for the current session.

```json
{}
```

Returns: `Original bytes / Bytes saved / Reduction %`

**Permission**: read-only, always allowed.

## Search Architecture

### Dual FTS5 Tokenizers

Context mode uses two FTS5 tokenizers in parallel:

1. **Porter stemmer** (`tokenize='porter unicode61'`): handles stemmed word search. Querying "running" matches "run", "runs", "running".
2. **Trigram** (`tokenize='trigram'`): handles substring/partial-identifier search. Querying "tool_out" matches "record_tool_output".

Results from both tokenizers are merged via **Reciprocal Rank Fusion (RRF)** with k=60.

### Fuzzy Correction

When both Porter and trigram searches return zero hits, context mode attempts
fuzzy correction:

1. For each query word (length ≥ 3), pull vocabulary candidates of similar length sharing a 3-character prefix
2. Compute Levenshtein distance to each candidate
3. Replace the word with the closest match
4. Re-run both searches with the corrected query

The vocabulary is built from indexed content (tokens of length 3-50, max 50 per event).

### Session Scoping

All searches are scoped to the current `session_id`. Subagents share the main
agent's session_id, so their indexed outputs are searchable by the main agent.

## Read Lifecycle Tracking

Context mode includes a read lifecycle tracker that detects:

- **Stale reads**: the file was edited since the last read. Old read data is wrong. Coaching message: `[Read lifecycle: N previous read(s) of this file are now STALE (file was edited — old data is wrong, KB wasted in context)]`
- **Superseded reads**: the same file section was re-read without any edit in between. Old data is redundant. Coaching message: `[Read lifecycle: M previous read(s) are SUPERSEDED (re-read without edit — old data is redundant, KB wasted)]`

When `write` or `edit` modifies a file, `mark_file_edited()` is called, which:
1. Adds the file to the edited-files set
2. Untracks the read path so the next read is fresh (not blocked by `deny_repeated_reads`)

The coaching messages are prepended to read output, helping the agent avoid
wasting context on redundant reads.

## Cache-Aware Compression Gating

When bash output is compressed, the system checks whether the compression
saves at least 15%. If savings are <15%, compression is **skipped** and the
raw output is returned instead.

Rationale: rewriting the tool result would bust the provider's prefix cache
for a negligible token saving. The small saving isn't worth the cache miss.

## SQLite Schema

```sql
-- Event tracking
events (id, session_id, timestamp, category, priority, source, content, meta)
events_fts (FTS5: content, source, category — Porter tokenizer)
events_fts_trigram (FTS5: content, source, category — trigram tokenizer)

-- Tool output externalization
tool_output (id, session_id, tool, path_or_cmd, raw_output, summary,
             bytes_original, bytes_saved, timestamp)
tool_output_fts (FTS5: raw_output, summary, path_or_cmd — Porter)
tool_output_fts_trigram (FTS5: raw_output, summary, path_or_cmd — trigram)

-- Compact snapshots
compact_snapshots (id, session_id, snapshot, created_at)

-- Fuzzy correction vocabulary
search_vocabulary (id, word, source)
```

The database uses WAL journaling with a 5-second busy timeout. All tables are
session-scoped — search results only include content from the current session.

## Event Categories and Priorities

| Category | Priority | When |
|----------|----------|------|
| `tool_call` | LOW | Any tool invocation |
| `tool_result` | LOW | Successful tool completion |
| `command_fail` | HIGH | Non-zero bash exit |
| `file_write` | LOW | write tool used |
| `file_edit` | LOW | edit tool used |
| `file_read` | LOW | read tool used |
| `compaction` | CRITICAL | `/compact` invoked |
| `permission_deny` | HIGH | Permission denied |

## Database Location

The SQLite database is at `<data_dir>/context.db`, where `<data_dir>` is
`$XDG_DATA_HOME/nehme-harness/` (or `~/.local/share/nehme-harness/` on Linux,
`~/Library/Application Support/nehme-harness/` on macOS).

## Disabling Context Mode

Build without the `context` feature:

```bash
cargo install nehme-harness --no-default-features --features loop,git-worktree,mcp,subagents,archmd,status-signals
```

This removes all four ctx tools, auto-indexing, compact snapshots, and read
lifecycle tracking.

## Tool Availability

| Tool | Main agent | Subagent | BTW agent |
|------|-----------|----------|-----------|
| `ctx_search`   | yes | yes | yes |
| `ctx_execute`  | yes | yes | yes |
| `ctx_retrieve` | yes | no  | yes |
| `ctx_stats`    | yes | no  | yes |

The BTW agent uses a **separate session_id** for its context store writes, so
its indexed outputs don't pollute the main session's `ctx_search` results.

## Configuration

Context mode has no user-tunable config keys. All thresholds are hardcoded
constants:

| Constant | Value | Purpose |
|----------|-------|---------|
| `EXTERNALIZE_THRESHOLD` | 100KB | bash/subagent output externalization threshold |
| `THROTTLE_REDUCE` | 3 | ctx_search calls before preview reduction |
| `THROTTLE_BLOCK` | 8 | ctx_search calls before throttling |
| `MAX_SNAPSHOT_BYTES` | 8KB | Compact snapshot size cap |
| Cache gating threshold | 15% | Minimum compression savings to apply |
